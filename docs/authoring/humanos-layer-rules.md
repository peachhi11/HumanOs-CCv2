# HumanOS Layer Rules

This document is the quick ownership test for HumanOS authoring and agent workflows.

Use it whenever a fact could belong to more than one layer.

## Ownership Test

Ask: would this remain true if the character moved into another story, setting, or scene?

| If the answer is... | Put it in... | Examples |
| --- | --- | --- |
| Yes, it is true of the person | Character or persona truth | attachment pattern, family history, values, appearance, fears, speech habits |
| Yes, it is true of the setting | World truth | university culture, corporate rules, social norms, established NPC roles, city geography |
| Only true of this campaign or pairing | Story binding | arranged marriage premise, shared secret, inciting betrayal, fixed relationship history |
| Only true now | Runtime truth | current room, latest argument, immediate objectives, current suspicion, unresolved scene beat |

## Character Or Persona Truth

Keep the stable kernel story-free.

A compact seven-module structure is:

1. Identity and presentation
2. Personality and values
3. Internal model and decision rules
4. Psychology: wounds, fears, defenses, contradictions
5. Relationships and attachment behavior
6. Intimacy, boundaries, and vulnerability, only when relevant
7. Speech and interaction style

State causal rules rather than plot facts.

Prefer "withdraws when care feels like obligation" over "withdrew after the lake-house argument."

## World Truth

World truth is durable external logic.

Include stable facts about:

- era
- place
- culture
- power structures
- institutions
- recurring locations
- supporting-cast routines
- setting rules

Do not put the protagonist's current emotional reading of an NPC into world truth.

## Story Binding

Story binding is a campaign's fixed premise.

Capture the premise and pressures that make this pairing or arc distinct:

- relationship status
- known shared history
- inciting event
- agreements or secrets
- public stakes
- current narrative direction

Story binding can attach otherwise-portable characters to a specific story, but it does not describe the live scene.

## Runtime Truth

Runtime truth is current and replaceable.

Use this shape:

- current situation
- current objectives
- current mindset
- known facts
- immediate constraints
- open threads or next pressure

Write it from the active character's perspective when that helps the model drive the scene, while preserving `{{user}}`'s freedom to respond, decide, and change direction.

## Audit Checklist

- Does the persona or character card work in a new setting without contradicting itself?
- Does the world section avoid personal plot history?
- Does the story section avoid minute-by-minute scene status?
- Can the runtime section be replaced without altering the other sections?
- Do roleplay instructions prohibit speaking, choosing, feeling, or acting for `{{user}}`?
- Does each recurring NPC have a reason to appear beyond filler?
- Does the build offer pressure and choice rather than a fixed ending?

## Compression Order

1. Remove repeated traits and duplicate facts.
2. Replace long descriptions with behavioral rules.
3. Retain conflicts, boundaries, attachment logic, and speech anchors.
4. Move optional history, setting detail, and side-character depth into lorebook-ready entries.
5. Verify the token cap after each major cut.

Never cut the layer labels that prevent contamination.
